Technische Referenz

Wie GridGen aufgebaut ist

GridGen ist eine abhängigkeitsfreie, rein clientseitige Web-App. Es gibt kein Framework und kein Backend – die App besteht nur aus index.html, style.css und gridgen.js, ergänzt durch eine CDN-Bibliothek zur PDF-Erzeugung. Ein kleiner statischer Site-Generator erstellt die lokalisierten Seiten.

Projektstruktur

GridGen/
├── index.html          # generated — English app shell
├── style.css           # app styles
├── gridgen.js          # all application logic
├── favicon.svg, og-image.png, robots.txt, sitemap.xml
├── nl/                 # generated — Dutch pages (/nl/, /nl/wiki/…)
├── wiki/               # generated — English docs + wiki.css
├── build.js            # static-site generator
└── site/               # generator SOURCE
    ├── layouts/        # base.html
    ├── partials/       # head.html
    ├── locales/        # en.json, nl.json — metadata + schema strings
    └── pages/<lang>/   # per-language page bodies
Generiert vs. Quelle. Das HTML im Repository-Stammverzeichnis sowie unter wiki/ und nl/ wird generiert – bearbeite die Templates in site/ und führe node build.js erneut aus; bearbeite nie die Ausgabe direkt.

Build & Sprachen

Jede Sprache wird aus einem Satz Templates plus einer sprachspezifischen JSON-Datei gerendert. Englisch wird im Site-Root ausgegeben; jede andere Sprache liegt unter /<lang>/. Der Generator schreibt außerdem die hreflang-Alternativen, den sprachspezifischen <title>/Meta- und JSON-LD-Inhalt, einen Sprachumschalter und eine mehrsprachige sitemap.xml.

Eine neue Sprache hinzuzufügen ist eine vierschrittige Änderung – Inhalte plus eine einzelne Konfigurationszeile:

  1. Füge site/locales/<lang>.json hinzu (Titel, Beschreibungen, Nav-Labels, Schema-Strings).
  2. Füge übersetzte Seiteninhalte unter site/pages/<lang>/ hinzu.
  3. Registriere den Sprachcode in build.js.
  4. Führe node build.js aus.

Zustandsmodell

Der gesamte App-Zustand liegt in einem einzigen einfachen Objekt. Jedes Steuerelement schreibt darin, und eine einzige render()-Funktion liest daraus – ein kleiner unidirektionaler Datenfluss ohne Framework.

const state = {
  image: null,        // the decoded HTMLImageElement
  imageName: '',      // used to name the exported file
  gridMode: 'cells',  // 'cells' | 'size'
  cols: 10, rows: 10, // for 'cells' mode
  cellSize: 50,       // for 'size' mode (px)
  color: '#000000',
  thickness: 2,
  opacity: 1.0,
  showLabels: true,
  labelStyle: 'chess',     // 'chess' | 'numeric'
  labelPosition: 'outside',// 'outside' | 'inside'
  origin: 'top-left'       // 'top-left' | 'bottom-left'
};

Jeder Eingabe-Listener ändert ein Feld und ruft render() auf. Es gibt kein virtuelles DOM und kein Diffing – der Canvas wird bei jeder Änderung einfach komplett neu gezeichnet, was für ein einzelnes Bild kostengünstig ist.

Rendering-Pipeline

render() zeichnet den gesamten Canvas in einer festen Reihenfolge neu. Das Zeichnen von hinten nach vorne sorgt dafür, dass spätere Ebenen natürlich über früheren liegen.

  1. Geometrie berechnen – Zellenbreite/-höhe und Spalten-/Zeilenanzahl aus dem aktiven Gittermodus ableiten.
  2. Canvas dimensionieren – Bildabmessungen plus oberen und linken Rand, wenn außen liegende Beschriftungen aktiviert sind (die Ränder, die die Spalten- und Zeilenbeschriftungen aufnehmen).
  3. Hintergrund weiß einfärben, dann das Bild mit dem Randversatz zeichnen.
  4. Gitterlinien zeichnen mit der gewählten Farbe, Stärke und Deckkraft.
  5. Beschriftungen zeichnen – entweder im äußeren Rand oder innerhalb jeder Zelle.
  6. Info-Leiste aktualisieren mit den aktuellen Zellen- und Bildabmessungen.

Gittergeometrie

Die beiden Gittermodi ergeben dieselben vier Werte – cols, rows, cellW, cellH – sodass dem Rest des Renderers egal ist, welchen Modus du gewählt hast:

if (state.gridMode === 'cells') {
  cols = state.cols;  rows = state.rows;
  cellW = img.width / cols;       // cells may be rectangular
  cellH = img.height / rows;
} else {
  cellW = cellH = state.cellSize; // square cells
  cols = Math.ceil(img.width / cellW);
  rows = Math.ceil(img.height / cellH);
}

Wenn außen liegende Beschriftungen aktiviert sind, reserviert GridGen einen Rand, der relativ zur Bildgröße skaliert, damit er mit der Auflösung mitskaliert:

const baseMargin = Math.max(36, Math.min(img.width, img.height) * 0.045);

Gitterlinien werden mit Math.min(i * cellW, img.width) auf die Bildgrenzen beschränkt, sodass eine unvollständige letzte Zelle im Modus Nach Größe nie über den Rand hinausgezeichnet wird.

Beschriftungsalgorithmus

Spaltenbeschriftungen im Schach-Stil verwenden eine bijektive Basis-26-Umrechnung – dasselbe Schema, das Tabellenkalkulationen für Spalten verwenden, sodass nach Z AA folgt:

function getColumnLabel(i) {
  if (state.labelStyle === 'numeric') return String(i + 1);
  let label = '', n = i;
  while (n >= 0) {
    label = String.fromCharCode(65 + (n % 26)) + label;
    n = Math.floor(n / 26) - 1;
  }
  return label;
}

Die Zeilennummerierung berücksichtigt die Ursprung-Einstellung, indem der Index umgekehrt wird, wenn der Ursprung unten links liegt:

const rowIdx = state.origin === 'top-left' ? i : rows - 1 - i;

Außen liegende Beschriftungen werden in den Randstreifen oben und links zentriert. Innen liegende Beschriftungen werden in der oberen linken Ecke jeder Zelle mit einem gestrichelten Halo hinter dem Fülltext gezeichnet, damit sie über komplexen Bildinhalten lesbar bleiben.

Kontrast & Lesbarkeit

Die Beschriftungsfarbe wird automatisch aus der wahrgenommenen Helligkeit der Gitterfarbe gewählt (Rec. 601-Gewichtung). Helle Gitterfarben erhalten dunkle Beschriftungen; dunkle Farben behalten ihren Farbton:

function isLight(hex) {
  const { r, g, b } = hexToRgb(hex);
  return (0.299 * r + 0.587 * g + 0.114 * b) > 186;
}

Bei innen liegenden Beschriftungen kehrt sich der Halo um – ein weißer Halo hinter dunklem Text oder ein schwarzer Halo hinter hellem Text – und gewährleistet so Kontrast unabhängig vom darunterliegenden Inhalt.

PDF-Export

Der Export verwendet den bereits gerenderten Canvas, sodass das PDF genau der Vorschau entspricht. Der Canvas wird in eine PNG-Daten-URL kodiert und auf einer A4-Seite platziert, deren Ausrichtung dem Seitenverhältnis des Bildes folgt:

const w = canvas.width, h = canvas.height;
const orientation = w > h ? 'landscape' : 'portrait';
const pdf = new jsPDF({ orientation, unit: 'pt', format: 'a4', compress: true });

// available area inside a 24pt page margin
const pageW = pdf.internal.pageSize.getWidth();
const pageH = pdf.internal.pageSize.getHeight();
const margin = 24;
const availW = pageW - 2 * margin, availH = pageH - 2 * margin;

// scale to fit, then center on the page
const scale = Math.min(availW / w, availH / h);
const drawW = w * scale, drawH = h * scale;
const x = (pageW - drawW) / 2, y = (pageH - drawH) / 2;

const imgData = canvas.toDataURL('image/png');
pdf.addImage(imgData, 'PNG', x, y, drawW, drawH);
pdf.save(`${baseName || 'grid-map'}-grid.pdf`);

Da der Quell-Canvas die nativen Pixelabmessungen des Bildes hat, bleiben die Gitterlinien beim Herunterskalieren auf die Seite scharf. Der Ausgabedateiname wird aus der hochgeladenen Datei abgeleitet, ihre Erweiterung wird entfernt und -grid angehängt.

jsPDF wird bei Bedarf geladen, nicht im Voraus. exportPDF() ist async und wartet auf einen kleinen Loader, der das CDN-<script> beim ersten Bedarf einfügt — im Hintergrund schon vorbereitet, sobald du ein Bild hinzufügst. Wer nie exportiert, lädt nichts vom CDN.

let jspdfPromise = null;
function loadJsPDF() {
  if (window.jspdf) return Promise.resolve();
  if (!jspdfPromise) {
    jspdfPromise = new Promise((resolve, reject) => {
      const s = document.createElement('script');
      s.src = JSPDF_SRC;            // the cdnjs UMD bundle
      s.onload = resolve;
      s.onerror = () => { jspdfPromise = null; reject(new Error('Failed to load jsPDF')); };
      document.head.appendChild(s);
    });
  }
  return jspdfPromise;              // memoized; retries after a failure
}

async function exportPDF() {
  await loadJsPDF();
  const { jsPDF } = window.jspdf;
  // … build the PDF (above) …
}

Datenschutzmodell

GridGen ist rein clientseitig. Das Bild wird mit dem FileReader des Browsers als Daten-URL gelesen und in ein Image-Element dekodiert – es wird nirgendwo übertragen:

const reader = new FileReader();
reader.onload = (ev) => { img.src = ev.target.result; };
reader.readAsDataURL(file);

Es gibt keine Analysen, keinen Server und keine Netzwerkanfrage außer dem Laden der jsPDF-Bibliothek von deren CDN — und das nur bei Bedarf. Alles andere läuft auf der Seite.

GridGen erweitern

Das Design mit einer einzelnen Logikdatei hält Änderungen lokal. Häufige Erweiterungen:

  • Neue Beschriftungsstile – füge einen Zweig zu getColumnLabel() und dem Beschriftungs-Zeichencode hinzu.
  • Exportformate – der Canvas kann mit canvas.toBlob() in PNG/JPEG umgewandelt werden für direkten Bild-Download.
  • Nicht-uniforme Gitter – der Geometrieblock ist die einzige Quelle der Wahrheit für die Zellengröße.
  • Voreinstellungen – das flache state-Objekt lässt sich sauber als JSON serialisieren für Speichern/Laden oder URL-Sharing.
